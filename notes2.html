While Loops (relevant .py starting here)

Loops, way to do things over and over

while loop:

while expression:
  block

As we saw with if statements, the block is a set of instructions
with if the block evaluates either 0 or 1 times depending on whether the test expression is True
with while the block continues to be evaluated as long as the test expression ('expression' above) is True

A simple example:

i = 0
while i < 10:
  print i
  i = i + 1

Important to note: after this loop, the new value of i is 10! While loop does not require a return statement!

Break Statements
break gives us a way to stop the loop while the test conditions are True, it tells to interpreter to jump out of the while loop and continue with any code that comes next

A simple example:

while test_expression:
  Python code goes here
  if break_test:
    break

As soon as break test is True we exit the loop

Debugging (relevant .py)

Bugs Happen!
Bugs are inevitable! Develop a strategy for getting rid of them.

Sometimes the bug causes the program to crash, in this case Python will give you a 'traceback' error, this will tell you where it crashed, including line number and function name, even if the error is not on that line, it is somewhere upstream from there

You can always Google the error message! e.x. TypeError: unsupported operand type(s) for +: 'int' and 'str'
in this case, you can't concatenate a number and a string, so you must use the str() function to 'stringify' the number (see .py)

When learning a new feature or language, you will often start with example code and then change it to do what you want it to

If that doesn't work...

  • 'comment out' your code
  • copy and paste the example code
  • run it to make sure the example does what it should
  • then modify in small steps until it does what you wanted
  • then compare to your original code to see if you can find the bug

All example code is not created equal! Sometimes it contains its own bugs and other times there are simply better examples to work from!

Bugs don't always cause programs to crash, these can be trickier to deal with!

If a function calculates an incorrect value, you often need to break it up into smaller pieces to find where it went wrong

One easy way to do this is to introduce print statements that output the variables where you can see them to check for weirdness

Keep and Compare Old Versions!
Comment out old version with hash mark (#) or turn it into a big string by putting ''' (triple quotes) around it, this is called a 'docstring' which is a multiline string (use either # or ''' and be consistent!)

Or copy code to different file and save it for your reference

Keeping old versions is useful when you ask for help because you can show 2-3 different things that you've tried

Version control systems like Git make this even easier

Debugging Recap

  1. Examine error messages
  2. Work from example code
  3. Make sure examples work
  4. Check (print) intermediate results
  5. Keep and compare old versions

Debugging isn't just finding and correcting errors, it is a really important skill and scientific process (i.e. it uses the scientific method!) Udacity has an entire course dedicated to debugging!

Comments are helpful because they allow you to compare what the code should be doing with what it actually does (i.e. it is a good idea to put the 'should' in comments as you go)

Commenting tips:

  • Don't comment 'obvious code' (i.e. code that is self explanatory)
  • Start functions with a comment, e.x.:

    def isLeapYear(year):
    # takes a number as input and outputs True if the number
    # represents a leap year and False otherwise

  • Keep comments up-to-date, change them as your code changes
  • Be concise

def median(a,b,c):
  if a >= b:
    if c <= a and c >= b:
    return c
  if c >= a:
    return a
  return b
  if c >= a and c <= b:
    return c
  if c >= b:
    return b
  else:
    return a

def bigger(a,b):
  if a > b:
    return a
  else:
    return b

def biggest(a,b,c):
  return bigger(a,bigger(b,c))

def median(a,b,c):
  big = biggest(a,b,c) # eliminates one number as median
  if big == a:
    return bigger(b,c)
  if big == b:
     return bigger(a,c)
  else:
    return bigger(a,b)

New vocab words RE: Lists:

  • mutation - supported by lists and not strings, we can change the value of a list after we've created it!

    For a string, concatenation does not change the string, it creates a whole new string:

    s = 'Hello'
    s = 'Yello'
    s = s + 'w'

    For lists:

    p = ['H','e','l','l','o']
    p = ['Y','e','l','l','o'] # creates a new list
    p[0] = 'Y' # mutates an element to a new value!

    If you introduce a new variable and then mutate it, it changes the value of BOTH lists (following from above):

    q = p
    q[4] = '!'

    Now q = p = ['Y','e','l','l','!']

    In other words: strings are not mutable! You can not change their value

    If an object is mutable, we have to worry about other variables that might refer to the same object (e.x. q and p above and in .py)

  • aliasing - two different ways to refer to the same object

    e.x. James Bond and 007 refer to the same person, if something happens to James Bond, it affects the state of both James Bond and 007

    If we have two variables names refering to the same object, changes to one change both!

    p = ['J','a','m','e','s']
    p = q
    p[2] = 'n' # this changes the value of both p AND q
    p = [0,0,7] # this also changes the value of both

When using a function the parameter and input are (at least temporarily) aliases:

def replace_spy(p):
  p[2] = p[2] + 1

spy = [0,0,7]
replace_spy(spy)
print spy

In this example p and spy refer to the same object inside the function

This is different from the following function:

def inc(n):
  n = n + 1
  return n

Removing the return statement from inc does not change the value of the input, only (temporarily) changes the value of n inside the function

Numbers are immutable!

Understanding a Problem...
means understanding the relationship between possible inputs and desired outputs

  1. What are the inputs?

  2. What are the outputs?
  3. Work through some examples by hand

    This might include thinking about what to do with invalid inputs

  4. Come up with a simple mechanical solution

    else if (year is not divisible by 100) then (it is a leap year)

  5. Don't optimize prematurely! Start with finding a simple and correct solution! Getting approximate answers at first is always a great confidence booster.
